# Serial Multi-AI Router Design

## Objective

Add `src/psi_agent/router/` as an OpenAI-compatible component between Session and multiple AI backends. For each new user task, the Router uses one routing backend to split the task into exactly three subtasks, assigns each subtask to a configured backend socket, executes the three subtasks serially with full workspace-tool access, and asks the same routing backend to aggregate the three subanswers. If any orchestration stage fails, the Router abandons the run and proxies the current Session request to a configured default backend socket.

The Router knows only socket addresses and capability descriptions. Provider and model configuration remains owned by each `psi-agent ai` service.

## Component Boundaries

The component chain is:

```text
Channel
  -> Session (conversation, ToolRegistry, tool execution)
  -> Router (planning, branch state, serial orchestration, aggregation, fallback)
  -> AI backend sockets
```

Session remains the only component that loads or executes workspace tools. Router receives OpenAI tool schemas, forwards them to the active subtask backend, relays tool calls to Session, and associates returned tool results with the active private branch. AI backends remain stateless OpenAI-compatible services whose actual models are fixed by their own startup configuration.

Router is an independent server exposing `POST /chat/completions` over Unix sockets, Windows Named Pipes, or HTTP URLs through the existing `psi_agent._sockets` helpers.

## Public Configuration

The Router dataclass has this conceptual interface:

```python
@dataclass
class Router:
    session_socket: str
    router_socket: str
    default_socket: str
    upstream: list[tuple[str, str]]
    max_tool_rounds: int = 10
    router_timeout: float | None = 60.0
    branch_timeout: float | None = None
    aggregate_timeout: float | None = None
    run_ttl: float = 1_800.0
    verbose: bool = False
```

The fields have the following meanings:

- `session_socket` is the address on which Router listens for Session requests.
- `router_socket` is one AI backend used for both three-subtask planning and final aggregation.
- `default_socket` is the ordinary AI backend that receives the current Session request after an orchestration failure.
- `upstream` is a `list[tuple[str, str]]`; each tuple is `(socket, description)`. It contains no model, provider, target ID, or API key.
- `max_tool_rounds` limits tool rounds independently for each subtask.
- the timeout fields bound planning, individual branch calls, and aggregation when non-`None`.
- `run_ttl` limits how long Router retains a run while waiting for Session to return tool results.

At least one upstream is required. Each tuple must contain two non-empty strings. Duplicate sockets are allowed; their descriptions are combined in the planning prompt. One upstream may serve any number of the three subtasks. `default_socket` may equal `router_socket` or an upstream, but it must not equal `session_socket`, which would recursively route a fallback request back into Router.

The CLI exposes Router as a top-level `psi-agent router` subcommand. Tyro receives `upstream` as its native `list[tuple[str, str]]` type rather than JSON strings. The implementation and tests must confirm the exact CLI token syntax generated by the installed Tyro version.

## Internal Session Identity

Router orchestration spans multiple Session requests whenever a subtask calls a tool. Session therefore adds stable internal metadata to every request sent to its AI socket:

```json
{
  "routing": {
    "session_id": "the-stable-session-id"
  }
}
```

The existing ordinary AI handler continues removing `routing` before contacting external providers. Router uses the session ID to locate one active run. The metadata never reaches planning, branch, aggregation, or default AI backends.

One Session may have at most one active routing run. A request that cannot be identified as the expected continuation of that run is treated as a state error and falls back instead of overwriting the run.

## Planning

For a new user task, Router sends the bounded conversation context and the socket-description catalog to `router_socket`. The planning backend must return JSON containing exactly three tasks:

```json
{
  "tasks": [
    {"subtask": "Collect and verify relevant facts", "socket": "http://127.0.0.1:8111"},
    {"subtask": "Analyze the facts and derive conclusions", "socket": "http://127.0.0.1:8112"},
    {"subtask": "Find omissions and counterexamples", "socket": "http://127.0.0.1:8112"}
  ]
}
```

Router validates the decoded JSON defensively:

- the root must be a dictionary;
- `tasks` must be a list of exactly three dictionaries;
- every `subtask` must be a non-empty string;
- every `socket` must exactly match a configured upstream socket;
- repeated socket choices are valid;
- unknown keys do not grant additional behavior.

Planner output does not use fuzzy model-name parsing and cannot introduce an unconfigured destination. An invalid plan triggers one structure-repair request to `router_socket`. A second invalid result triggers the default fallback path.

## Private Branch State

Each active run contains exactly three serial branches:

```python
@dataclass
class BranchState:
    subtask: str
    socket: str
    messages: list[dict[str, Any]]
    status: BranchStatus
    tool_rounds: int = 0
    final_answer: str = ""


@dataclass
class RoutingRun:
    run_id: str
    session_id: str
    original_messages: list[dict[str, Any]]
    tools: list[dict[str, Any]]
    branches: tuple[BranchState, BranchState, BranchState]
    current_branch: int = 0
    pending_tool_calls: dict[str, ToolCallOrigin] = field(default_factory=dict)
    last_active_at: float = 0.0
```

The state progression is strictly serial:

```text
planning
  -> branch 1 running <-> branch 1 waiting for tools
  -> branch 1 completed
  -> branch 2 running <-> branch 2 waiting for tools
  -> branch 2 completed
  -> branch 3 running <-> branch 3 waiting for tools
  -> branch 3 completed
  -> aggregating
  -> completed
```

Branch 2 is not contacted before branch 1 completes, and branch 3 is not contacted before branch 2 completes. There is never more than one active branch AI request.

All three branch backends receive the same complete `tools` schema list. They choose tools independently based on their assigned subtasks.

## Branch Context and Serial Knowledge Transfer

Branch 1 receives the original task and its own subtask. Branch 2 additionally receives branch 1's final subanswer. Branch 3 additionally receives branch 1 and branch 2's final subanswers.

Later branches do not receive earlier branches' private reasoning, tool-call arguments, raw tool results, or intermediate assistant messages. Only final subanswers cross the branch boundary. This preserves useful serial collaboration while controlling context size and preventing accidental propagation of noisy or sensitive tool output.

Each branch has a private OpenAI message history. Router forwards the original request's generation options where compatible, replaces `messages` with the private branch history, supplies the original complete `tools` list, removes `routing` and `model`, and requests streaming. Router never sets a model field based on routing: the AI service behind the selected socket owns its model.

## Tool Calls and Session Round Trips

When the active branch finishes with `tool_calls`, Router stores the assistant message in that branch's private history and rewrites every upstream tool-call ID to a run-global ID. The mapping records the active branch and original upstream ID. Router then streams those calls to Session with `finish_reason="tool_calls"` and waits for the next Session request.

Session executes the calls through its existing ToolRegistry and records the tool results in its formal conversation. On the continuation request, Router:

1. resolves the run by `routing.session_id`;
2. validates that every newly returned tool result matches a pending rewritten ID;
3. restores the original upstream ID inside the private branch history;
4. changes the branch from waiting to running;
5. calls the same branch socket again.

A branch may perform several tool rounds and may call different tools on every round. It advances to the next branch only after receiving `finish_reason="stop"`. Reaching `max_tool_rounds`, receiving unrelated tool results, or observing an impossible state triggers fallback.

When Router exposes a branch tool call to Session, the assistant content includes a short stable description such as `Processing subtask 1: ...`. It does not expose the branch's private reasoning. This gives a later default backend enough context to interpret already-recorded tool results if fallback becomes necessary.

## Aggregation

After all three branches complete, Router calls `router_socket` again with a separate aggregation prompt containing:

- the original task;
- each subtask;
- each final subanswer.

Aggregation receives no tools and no private branch tool histories. It produces the single final natural-language answer. Router exposes that answer as OpenAI-compatible single-choice SSE content followed by `finish_reason="stop"`, then immediately deletes the run.

The planning and aggregation roles share one socket and therefore one backend-defined model, but they use separate prompts and stateless conversations.

## Default Fallback

Any unrecoverable orchestration failure terminates the three-subtask run and switches the current request to `default_socket`. Failures include:

- planning or plan-repair failure;
- upstream HTTP or SSE failure;
- zero usable result or multiple choices where one is required;
- `finish_reason="error"`;
- branch timeout or tool-round exhaustion;
- missing, expired, inconsistent, or unresolvable run state;
- malformed or unrelated tool results;
- aggregation failure.

Before fallback, Router deletes the active private run. It then proxies the current Session request to `default_socket`, retaining `messages`, `tools`, and unknown passthrough parameters, removing internal `routing` and `model`, and preserving the default backend's OpenAI-compatible SSE stream. The default backend may answer directly or enter the normal Session tool loop.

Fallback occurs at most once per incoming Router request. Failure of `default_socket` is returned as HTTP 502 if the downstream response has not been prepared, or as an SSE error chunk with `finish_reason="error"` after streaming has begun. Router never routes a default failure back through planning.

If branch tools were already executed, the Session conversation supplied to the default backend contains their calls and results, including the short subtask description emitted with the tool calls. Router does not repeat those tools or expose additional private branch history.

## Protocol and Streaming Rules

Router accepts and emits the existing OpenAI Chat Completions HTTP/SSE dialect. It preserves the project invariant of exactly one choice per non-heartbeat chunk. Zero-choice heartbeat chunks are skipped; multiple choices are treated as orchestration failures.

Every SSE chunk crossing Router's upstream or downstream boundary is logged at DEBUG without logging secrets. HTTP errors before `response.prepare()` use the standard OpenAI-style error object. Errors after preparation use a ChatCompletionChunk with `finish_reason="error"`.

The final aggregation may be streamed through as it arrives. Planning and branch final text are internal; branch tool calls are streamed to Session only when Session action is required.

## Cancellation, Cleanup, and Expiry

Every upstream async generator or response stream is explicitly closed when iteration ends early or cancellation occurs. Cleanup across `await` is protected by `anyio.CancelScope(shield=True)`. Cancellation is not caught as `Exception` and is never swallowed.

Client disconnect or service shutdown cancels the current upstream call and deletes the affected run. Router shutdown closes its aiohttp runner and clears all active runs. Expired runs waiting for tool results are removed according to `run_ttl`, either by bounded background cleanup or on access; the implementation must not leave an uncancellable background task.

Startup failure and normal shutdown both perform shielded cleanup. Socket files remain subject to the project's existing deliberate no-auto-unlink policy.

## Logging

Logging follows project conventions:

- INFO: service lifecycle, run creation/completion, branch transitions, fallback destination;
- DEBUG: sanitized plan, socket choice, tool-call ID mapping, every SSE chunk, lock acquisition/release;
- WARNING: recoverable orchestration failures and fallback activation;
- ERROR: default-backend failure and unrecoverable server errors.

Logs never include API keys, full tool results, or unbounded message bodies. Socket addresses and short subtask summaries may be logged.

## Files and Responsibilities

The implementation should use focused modules under `src/psi_agent/router/`:

- `__init__.py`: public exports;
- `entry.py`: `Router` dataclass, parameter validation, lifecycle orchestration;
- `protocol.py`: branch/run dataclasses, enums, and typed parsed results;
- `client.py`: socket-aware OpenAI HTTP/SSE client shared by planning, branches, aggregation, and fallback;
- `planner.py`: planning prompt construction, strict JSON parsing, one repair attempt;
- `orchestrator.py`: serial branch state machine, tool-call mapping, aggregation, fallback decisions;
- `server.py`: aiohttp request handler and downstream SSE encoding;
- `prompts.py`: package-owned planning, repair, branch, and aggregation templates.

Session changes are limited to attaching internal stable `routing.session_id` metadata. CLI changes add Router to the top-level union. Documentation must describe component placement, socket-only configuration, tool behavior, and fallback semantics.

## Test Strategy

Development follows test-first red-green-refactor cycles.

Unit tests cover:

- native `list[tuple[str, str]]` CLI/config shape;
- empty fields, missing upstreams, and recursive default rejection;
- duplicate socket acceptance and description merging;
- strict three-task plan parsing and one repair attempt;
- repeated socket assignment across tasks;
- rejection of unconfigured upstream sockets returned by Planner;
- serial branch ordering;
- branch 2 receiving only subanswer 1 and branch 3 receiving only subanswers 1 and 2;
- exclusion of earlier reasoning, tool arguments, and raw tool results;
- tool-call ID rewrite and restoration;
- multi-round tools within one branch;
- tool-round limit and mismatched tool-result fallback;
- aggregation input and omission of tools;
- run completion, expiry, failure, and cancellation cleanup;
- HTTP versus SSE error behavior;
- single default fallback with no recursive retry.

Integration tests use several inline aiohttp mock sockets with per-request responses. The primary integration scenario verifies:

1. `router_socket` returns three planned tasks;
2. branch 1 calls a tool and completes after receiving its result;
3. branch 2 starts only afterward, sees subanswer 1, calls a different tool, and completes;
4. branch 3 starts only afterward, sees subanswers 1 and 2, and completes without a tool;
5. `router_socket` receives all three final subanswers for aggregation;
6. Session receives a valid single-choice final SSE response;
7. every task request reaches its exact planned socket.

Additional integration scenarios verify planner, branch, state-recovery, and aggregation failures each proxy the current request to `default_socket`, including a fallback that continues using tools already recorded in Session history.

Final verification includes the focused Router and Session tests followed by:

```text
uv run ruff check .
uv run ruff format --check .
uv run ty check
uv run pytest -v
uv run psi-agent router --help
```

## Deliberate Non-Goals

This feature does not execute workspace tools inside Router, expose model/provider/API-key configuration in Router, run branches concurrently, persist private branch histories to disk, share raw tool results between branches, allow Planner to invent sockets, or silently aggregate incomplete branch results.
