psi-agent 国内通道接入调研

更新日期：2026-06-16 ｜ 范围：QQ、微信、飞书、钉钉 ｜ 目标：明确实现顺序和风险边界

当前方案：飞书、钉钉优先走官方开放平台事件/机器人能力，已经有 psi-agent adapter 和本地 mock E2E。微信、QQ 先落在“外部 bridge -> psi-agent 标准化 webhook -> bridge reply”的边界上，避免把个人号逆向协议放进 psi-agent。

命名边界：psi-agent gateway 是本项目自己的 profile runtime，只负责统一启动 AI backend、session 和多个 channel adapter；它不是某个外部项目，也不等于微信/QQ 的登录桥。

当前代码链路

平台或外部 bridge
  -> src/psi_agent/channel/platform.py adapter
  -> psi-agent session socket
  -> AI backend / model
  -> adapter.send_reply()
  -> 平台或 bridge 发回聊天窗口

微信/QQ bridge 契约

psi-agent 侧不负责登录微信或 QQ，而是接收外部 bridge 转发的标准化消息。最小入站 payload：

{
  "type": "message",
  "message": {
    "conversation_id": "room-1",
    "user_id": "user-1",
    "message_id": "msg-1",
    "text": "hello"
  },
  "reply_url": "https://bridge.example/reply"
}

最小出站 payload：

{
  "conversation_id": "room-1",
  "user_id": "user-1",
  "text": "reply",
  "in_reply_to": "msg-1"
}

鉴权支持 Authorization: Bearer <secret> 或平台专用 bridge secret header。

飞书

  • 入站：飞书开放平台 Events API，重点事件是 im.message.receive_v1。
  • 出站：消息回复 API /open-apis/im/v1/messages/{message_id}/reply，或按 chat_id 主动发送消息。
  • psi-agent adapter 已支持直接传 tenant_access_token，也支持 app_id/app_secret 自动换 token。

钉钉

  • 当前 HTTP 方案：应用机器人收到消息后回调到 psi-agent，payload 中的 sessionWebhook 用于回复。
  • 真实使用中群聊通常要求 @ 机器人后才触发，这是钉钉机器人触发规则，不是 psi-agent 逻辑问题。
  • 后续需要支持 Stream 模式，用长连接减少公网 webhook 配置和回调稳定性问题。
