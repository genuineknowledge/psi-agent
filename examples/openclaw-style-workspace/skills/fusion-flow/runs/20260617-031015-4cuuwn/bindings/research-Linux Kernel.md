# Linux Kernel (torvalds/linux) 社区分析报告

开始前有个关键前提需要说明：torvalds/linux 在 GitHub 上是一个**只读镜像**，不是实际的开发仓库。Linux 内核的开发完全不走 GitHub 的 Issue/PR 流程，而是基于**邮件列表（LKML 及各子系统列表）+ 邮件补丁（patch）+ 维护者层级树**的模式。因此本报告中的"Issue"和"PR"需要映射到内核社区的真实工作流，否则数据无意义。下面的分析按这个真实流程展开。

## 1. Issue 解决周期

内核没有集中式的 issue 跟踪器作为主流程。bug 主要通过三个渠道流转：邮件列表报告、[bugzilla.kernel.org](https://bugzilla.kernel.org)，以及近年来子系统逐步采用的工具。

- **平均解决时间（估算）**：无统一数字。Bugzilla 上大量 bug 长期处于 open 状态，社区普遍承认 Bugzilla 的实际跟踪效果较弱，很多报告无人响应。安全类问题（CVE）则相反，关键漏洞通常在数天到数周内修复并通过 stable 树下发。依据：内核安全团队的公开处理流程和 stable 发布节奏。
- **典型生命周期**：报告 → 抄送相关子系统维护者/列表 → 维护者或贡献者讨论并复现 → 提交修复补丁 → 补丁评审合入对应子系统树 → 随合并窗口进入主线 → 通过 stable 回合到旧版本。很多 bug 不经过任何正式 tracker，直接在列表里以补丁形式"被修复"。
- **SLA / 响应承诺**：常规 bug **没有 SLA**。唯一接近承诺的是安全披露流程：通过 security@kernel.org 报告的漏洞有明确的协调披露窗口（通常最长 7 天的禁运期约定）。
- **长期未解决占比（估算）**：Bugzilla 上未关闭 bug 占比很高（我无法给出精确百分比，避免编造）。这更多反映"Bugzilla 不是主战场"，而非问题真的无人处理——很多实际修复发生在邮件列表，不会回写到 Bugzilla 关闭。

## 2. PR 合并率

对应内核的"补丁 + pull request（维护者向 Linus 发起的子系统拉取请求）"模型。这里需区分两个层级：开发者投递的补丁，和维护者汇总后的 pull request。

- **合并率（估算）**：
  - 维护者发给 Linus 的 pull request：合并率**极高**，接近 95%+，因为这些已经过子系统层层评审。Linus 偶尔会拒绝或要求重做，但属少数。
  - 开发者投到邮件列表的原始补丁：合并率**低得多**，大量补丁需要多轮修订（v2、v3……v10+很常见），首版直接合入的比例不高。无精确统计，谨慎估计原始投递补丁的最终合入率在中等偏低区间。
- **典型审查周期**：差异巨大。简单 fix 可能当天合入维护者树；复杂特性（如新调度器、文件系统改动）可能历经数月甚至数年、几十个版本迭代。
- **审查流程特点**：
  - 不依赖 GitHub CI 门槛，但有 `checkpatch.pl` 风格检查、子系统专用测试（如 KUnit、selftests）、以及 Intel 0-day、syzbot 等自动化测试机器人。
  - 评审基于 `Reviewed-by`、`Acked-by`、`Tested-by`、`Signed-off-by` 等 trailer 标签构成的信任链，维护者拥有最终决定权（"benevolent dictator" 层级模型）。
  - 通常至少需要对应子系统维护者的认可才能进树。
- **被拒绝/关闭的主要原因**：不符合代码风格、缺少 `Signed-off-by`(DCO)、设计不被维护者接受、破坏 ABI/向后兼容、未拆分成合理的补丁系列、缺乏测试、或单纯因为投递后无人跟进而"沉底"（silent drop 是常见的非正式拒绝方式）。

## 3. 社区治理特点

- **维护者团队规模与工作方式**：采用**金字塔式信任委托**。Linus Torvalds 是最终集成者，下面有 Greg Kroah-Hartman 等顶级维护者，再往下是约 **1000+ 名子系统维护者**（见内核源码树中的 `MAINTAINERS` 文件，这是精确数据来源）。每个维护者管理自己的 git 树，向上游发 pull request。
- **发布周期（精确）**：节奏非常规律。每个主线版本 = **2 周合并窗口 + 约 7 个 -rc 候选版本**，整体约 **9–10 周一个版本**。每年约 6–7 个主线版本。此外有 longterm（LTS）维护版本，维护期通常 2–6 年。
- **贡献者 onboarding 难度**：**偏高**。需要掌握 `git send-email`、邮件礼仪、补丁拆分、DCO 签署、子系统流程差异等。文档完善（`Documentation/process/` 下有完整指引，如 `submitting-patches.rst`），且有 first-patch 引导和 outreach 项目，但邮件工作流对习惯 GitHub PR 的新人门槛明显。
