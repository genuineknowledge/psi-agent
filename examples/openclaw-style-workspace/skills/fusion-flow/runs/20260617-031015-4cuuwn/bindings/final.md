## 💡 关键发现

- **发布节奏与项目活力正相关**：Rust（6 周）、Linux（9–10 周）、VS Code（月度）节奏越规律，处理节律越稳，综合评级越高。
- **"公司主导"显著压低外部 PR 合并率**：React、VS Code、TensorFlow 核心改动多走内部系统，外部贡献集中在文档/小修复。
- **GitHub 原始统计对部分项目失真**：Linux 是只读镜像（开发走邮件列表），TensorFlow 用 Copybara 同步致 PR 多显示 closed 而非 merged，二者合并率不可直接比较。
- **无项目提供正式 SLA**：但普遍用 stale-bot 自动关闭沉寂条目，人为压低 open 数，掩盖了真实长尾。
- **自动化门禁是共性**：CLA + 强制 CI 已成大型项目标配。

## 🏅 最佳实践总结

- **质量门禁最佳：Rust 的 bors**——合并前对每个 PR 单独跑全套 CI，保证 master 始终可构建。
- **分流响应最佳：Kubernetes SIG + VS Code triage**——领域归口与 1–3 天初步响应，规模化处理体量。
- **发布纪律最佳：Rust / Linux**——固定周期让贡献者可预期。
- **启示**：稳定的发布节奏 + 自动化质量门禁 + 清晰的领域分流，是大型项目可扩展协作的三大支柱；公司主导项目宜明确区分"内部 PR"与"外部 PR"统计，并主动降低外部核心贡献门槛。
