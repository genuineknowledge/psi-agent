# HTTP/3 (QUIC) 实时通信延迟分析

## 1. 协议机制对延迟影响
- **0-RTT**：重连复用会话票据，应用数据随首包发送，省 1 个 RTT。
- **无队头阻塞**：流级独立，单流丢包不阻塞其余流（解决 TCP 层 HOL）。
- **连接迁移**：以 Connection ID 标识，IP 变更不重握手。
- **TLS 1.3 内置**：传输+加密握手合并，1-RTT 完成。

## 2. 延迟数据
| 指标 | HTTP/3 | HTTP/2(TCP+TLS) |
|---|---|---|
| 首次握手 | 1-RTT | 2-3 RTT |
| 重连 | 0-RTT | 1-2 RTT |
| TTFB | 较 H2 低约 100-300ms（弱网更明显，Google/Cloudflare 实测）|—|

- **高丢包(>1%)**：Fastly/Cloudflare 报告丢包 2% 时 H3 尾延迟(p99)较 H2 改善可达 ~30%，因独立流避免连带重传等待。
- **连接迁移**：WiFi→5G 切换，H3 避免重建连接，省 ~1 个握手 RTT（可达数百 ms）。

*来源：Cloudflare、Fastly、Google QUIC 论文 (SIGCOMM'17)。具体数字随网络条件波动，需自测验证。*

## 3. 实时场景适配
- **消息推送**：0-RTT 重连近零延迟，适合频繁断连移动端。
- **直播弹幕**：万级并发流无 HOL，p99 更稳。
- **游戏**：QUIC DATAGRAM 扩展(RFC 9221)提供不可靠传输，避免重传抖动。
- **视频会议**：可插拔拥塞控制（BBR/CUBIC 可选），适配实时码率。

## 4. 协议对比
- **vs HTTP/2**：弱网/丢包收益显著；理想网络差异小。
- **vs WebTransport**：基于 H3，支持双向流+数据报，是浏览器实时通信未来方向（替代部分 WebSocket）。
- **部署**：约 30%+ 网站支持(W3Techs)；UDP 可能被中间设备限速/封锁，需 TCP 回退。

## 5. 最佳实践
- **选 H3**：移动端、弱网、高丢包、频繁重连场景。
- **调优**：实时选 BBR；0-RTT 仅用于幂等请求（防重放）；启用迁移；监控 UDP 阻断率并保留回退。
