# WebSocket 实时通信延迟分析

## 1. 协议机制
全双工长连接免去重复握手；帧头最小 2B（小载荷开销＜HTTP 数百字节头）；单连接内无 HTTP 队头阻塞，但仍受 TCP 字节流顺序约束；升级握手仅首次产生 1×RTT（+TLS 1.3 1-RTT）。

## 2. 延迟数据（典型值，注明来源）
| 项 | 数值 | 来源 |
|---|---|---|
| 建连(TLS1.3,同区域) | ~2×RTT≈20–40ms | Cloudflare 博客实测区间 |
| 单消息 RTT 64B/1KB/16KB | ~0.5/0.6/1.2ms(LAN) | 公开 ws benchmark 量级 |
| 1000 msg/s p50/p95/p99 | ~0.6/1.5/3ms(同城) | 同上量级估算 |
| 尾延迟 | 持续大流量下受 TCP 拥塞窗/重传放大，p99 可升至 10ms+ | Nagle/拥塞控制机理 |

注：以上为公开测试常见量级，跨网络受 RTT 主导，非权威基准；精确值需自测。

## 3. 实时场景
- 聊天：事实标准，端到端 50–150ms 体感无感。
- 协作文档：传输非瓶颈，OT/CRDT 合并计算占主导。
- 游戏：ws 走 TCP，丢包触发重传致卡顿；竞技类仍首选 UDP/WebRTC。
- 金融行情：亚毫秒仅限同机房内网；公网受 RTT 限制不可行。
- 物联网：2B 帧头省带宽，优于 HTTP 轮询。

## 4. 协议对比
- vs SSE：SSE 单向，回传需另开 HTTP；ws 双向省 1 次建连往返。
- vs gRPC streaming：gRPC 适合强类型微服务内部流；浏览器端 ws 更通用。
- vs WebTransport：基于 HTTP/3+QUIC，支持不可靠/多流，消除 TCP 队头阻塞，是低延迟下一代方案。

## 5. 最佳实践
- 心跳 ping/pong 间隔 20–30s，过频增负载、过疏致 NAT 超时重连（重连=数十 ms 抖动）。
- permessage-deflate：大文本省带宽，但压缩增 CPU 与微秒级延迟；高频小消息建议关闭。

（核心结论：公网延迟由 RTT 主导，ws 优势在省握手与双向；极致低延迟选 WebTransport/UDP。）
