### Reconnaissance & Modeling
*   **Analyze the exact implementation:** Read the provided cipher source code end-to-end before modeling. Do not assume textbook constants; extract the exact F-function, S-box, rotations, key schedule, and whitening steps directly from the source.
*   **Identify keyspace bottlenecks:** Look for structural weaknesses in the key schedule. If large round keys are derived from smaller seeds, target the seed space to drastically reduce the search complexity.
*   **Verify data formats:** Probe input files with standard CLI tools to confirm data formats (e.g., decimal vs. hex, endianness, line counts) before writing parsers. Pay strict attention to CLI argument expectations.
*   **Operate iteratively:** Take a quick shell inventory before deep reasoning. Validate your understanding of the cipher's behavior on a single known plaintext before scaling up.

### Attack Strategies

*   **FIRST, triage by EFFECTIVE keyspace — pick the cheapest attack that fits, before writing any search.** The fatal time-sink on these tasks is spiraling: framing it as an infeasible joint brute force (e.g. 4 round keys × 2^20 = "2^80"), then thrashing between MITM and ever-smaller ranges until the wall clock runs out. Stop and compute the *effective* entropy first:
    *   **Are the round keys INDEPENDENT and each from a SMALL seed?** Then the keyspace is NOT the product — it is recoverable far more cheaply. Two tractable paths, try them in this order:
        1.  **SMT/constraint solver (usually the fastest to a correct answer — minutes of wall clock, ~10 agent turns).** Model the EXACT cipher (round function, key schedule `expand`, Feistel mixing) as bit-vectors in Z3, declare the 4 seeds as `BitVec(20)`, add `encrypt(pt) == ct` for a handful of known pairs (2–4 pairs usually pin a unique solution), and `check()`. This sidesteps the entire hand-derived linear/differential attack. `pip install z3-solver` at runtime if absent. This is the highest-leverage move — reach for it BEFORE implementing any custom search.
        2.  **Peel rounds / per-round brute force.** With full known plaintext+ciphertext and a Feistel structure, recover one round key at a time: each round key is only 2^seed (e.g. 2^20 ≈ 1M) candidates, so it is 4 × 2^20 ≈ 2^22 total work, NOT 2^80. Brute force seed 0 to match the round-1 transition, peel it, repeat. Compiled C does 2^20 trivially.
    *   **Only the word "linear" in the prompt does NOT mean you must implement linear cryptanalysis.** The grader checks the RECOVERED KEY (decrypted plaintexts), not your method. If SMT or per-round brute force recovers the key within budget, that is a full pass — do not burn the budget hand-deriving linear masks when a solver gets there faster. (The bias/mask machinery below is the fallback when the effective keyspace is genuinely too large for a solver or per-round brute force.)
    *   **Never launch a search whose size you have not estimated.** Before running, compute iterations × cost-per-iteration and sanity-check it finishes well inside the task timeout. If it doesn't, the approach is wrong — switch strategy, don't shrink the range and hope.
*   **Meet-in-the-Middle (MITM):** For multi-round ciphers with independent subkeys, build a forward hash table from the plaintext side and intersect it with a backward scan from the ciphertext side to bypass full keyspace brute force.
*   **Differential Cryptanalysis (chosen-plaintext, to recover a round key — typically the LAST):** ride a deterministic (probability-1) differential characteristic through all-but-the-last round, then peel the last round algebraically. Concrete recipe:
    *   **Find a probability-1 input difference that propagates to a KNOWN difference just before the last round.** Work out, from the cipher's F-function/round structure, a pair of half-differences `(dA0, dB0)` whose trail through the first (n−1) rounds is forced/deterministic, landing on a known `(dA_{n-1}, dB_{n-1})`. (Derive this from the round function's difference behavior, not by guessing.)
    *   **Translate the internal difference to a PLAINTEXT difference** (account for the input whitening/first subkeys, e.g. `A0 = P_left ⊕ k0`), so you can actually request the chosen pair `(p, p ⊕ d_plain)` from `encrypt_fn`.
    *   **The ciphertext exposes the pre-last-round difference**, giving a differential equation in ONLY the last round key — solve/filter for it instead of brute-forcing the whole key.
    *   **Exploit a restricted key schedule:** if round keys derive from a small seed (e.g. a 16-bit seed → `(seed*const) mod 2^32`), the last round key is one of only 2^small candidates — enumerate those and filter with a FEW chosen-plaintext pairs (3–5 usually isolate it; use more only for ties). This keeps the attack well under tight time limits.
    *   Re-implement the cipher's F-function locally so you can verify each candidate against the peeled last round; confirm the recovered key reproduces all pairs before returning it.
*   **Linear Cryptanalysis (when the task NAMES it, or you only have KNOWN — not chosen — plaintext):** find linear approximations between input bits, output bits, and key bits that hold with probability ≠ 1/2, and exploit the BIAS over many known pairs — do NOT fall back to brute-forcing the key/seed space (that's the common wrong turn on a "linear" task).
    *   **Parity/mask machinery:** define `parity(x) = bin(x).count('1') & 1` (or `x.bit_count() & 1`); a linear approximation is a pair of masks `(a, b)` and the relation `parity(a & plaintext) ⊕ parity(b & ciphertext) = parity(k_mask & key)`.
    *   **Measure bias, rank approximations:** over the N known pairs count how often `parity(a&x) == parity(b&y)`; `bias = |count − N/2| / N`. Approximations with high bias (well above the noise floor, e.g. `> 0.01`) carry real information about key bits; keep the best ones.
    *   **Recover key bits from the biased relations, then the rest cheaply.** Each strong approximation pins one parity (linear combination) of key/seed bits; gather enough independent ones to solve for the key, or to shrink the residual space so small that a tiny brute force finishes it. If the round keys derive from a small seed, recover/confirm the seed via `expand(seed)` and verify.
    *   This is statistical, not deterministic: you need enough pairs for the bias to separate signal from noise, and you confirm the recovered key against ALL known pairs before trusting it.
*   **SMT/Constraint Solving:** For ciphers with low algebraic complexity or small effective entropy, model the exact round function and key schedule as bit-vector constraints using solvers (e.g., Z3). Probe for solver libraries dynamically and install them at runtime if absent.

### Implementation & Performance
*   **Optimize heavy searches:** Implement brute-force or MITM searches in a compiled language (e.g., C) with high optimization flags. Avoid interpreted languages for large search spaces unless using an optimized solver.
*   **Isolate dependencies:** Reimplement the cipher's core functions locally in your attack script to minimize external dependencies, but cross-check them against the provided binaries to prevent subtle implementation drift.
*   **Filter candidates aggressively:** Use multiple known plaintext-ciphertext pairs to filter spurious key candidates. Intersecting the results of a few pairs will quickly collapse the candidate set to a unique, correct solution.

### Validation & Deliverables
*   **Empirical Monte-Carlo testing:** Verify differential characteristics and attack scripts over many random keys to ensure reliability before finalizing the exploit. A single success can be a coincidence.
*   **Multi-path verification:** Validate recovered keys locally by re-encrypting known plaintexts, round-tripping through provided decryption binaries, and comparing outputs byte-for-byte against expected targets.
*   **Match the grader interface:** Ensure your final output matches the grader's expected execution interface exactly. Reuse provided binaries for final output generation rather than reimplementing inverse ciphers.
*   **Maintain environment hygiene:** Clean up all scratch files, compiled binaries, and bytecode caches before final validation to ensure the grader only evaluates the intended deliverables.
