### Before fitting: verify what the axes physically ARE
The most damaging error in instrument-data tasks is fitting in the wrong x-axis
representation — the fit "converges" but to a peak an order of magnitude off.
*   **Sanity-check the x-axis range against the expected unit.** If a task asks for peaks at values in one range (e.g. a material's known spectral features) but the file's first column spans a very different range, the column is probably in a *different physical quantity* than the answer wants, and you must convert it.
*   **Know the common instrument conversions and try them explicitly:**
    *   Wavelength (nm) ↔ wavenumber (cm⁻¹): `wavenumber = 1e7 / wavelength_nm` (since 1 cm = 1e7 nm). Spectrometers frequently record wavelength while the analysis wants wavenumber/Raman-shift. A decreasing wavelength column maps to an increasing wavenumber column.
    *   Raman shift = laser_wavenumber − scattered_wavenumber when an absolute axis is given.
    *   Pixel/channel index → physical axis via a stated calibration (linear `a*i+b` or polynomial). Detector bin numbers are NOT the physical axis.
    *   Energy (eV) ↔ wavelength (nm): `eV = 1239.84 / nm`.
*   **Confirm the conversion with a KNOWN feature, don't assume it.** After converting, check that a feature you can identify lands where domain knowledge says it should (a material's characteristic line, a calibration peak). If a transform makes the known features line up and the raw axis does not, the transform is right. This validation — not memorizing one constant — is the transferable skill.
*   **A fitted center that is off by ~an order of magnitude is almost always an axis/unit problem, not a fitting-algorithm problem.** Do not "fix" it by inventing a linear calibration to force the number; find the correct physical transform.

### Parse the file format defensively
*   **Locale-formatted numbers:** European data uses comma as the decimal separator and may use tab or semicolon as the field separator (`"1648,72\t5950,92"`). Replace `,`→`.` per field before `float()`; do not split on the comma.
*   **Headers / metadata lines / blank lines:** skip non-numeric leading lines; detect the data block by lines that parse as numbers.
*   **Column order is not guaranteed:** confirm which column is the independent axis (often monotonic) and which is intensity (non-monotonic, has the peaks).
*   **Probe libraries first:** if `numpy`/`scipy` may be missing, check and `pip install` them, or fall back to a pure-Python least-squares (closed-form for the linear parameters).

### Fit each peak in a local window
*   **Window around each target peak** rather than fitting the whole spectrum at once — neighboring peaks, substrate signals, and background slope corrupt a global fit. Open a window bracketing the expected center (informed by the converted-axis sanity check) and fit inside it.
*   **Make the window wide enough to capture the wings and flat baseline on BOTH sides — a too-narrow window is a top cause of a "close but just outside tolerance" fit.** A Lorentzian has long tails; if the window only spans the peak core, the fitter cannot separate the tail from the background, so it *underestimates the offset and inflates γ (width) to compensate* (and vice-versa). Aim for a window that extends several half-widths past the peak on each side so each edge sits on genuine baseline. If γ or offset land just outside tolerance while x0 and amplitude are fine, widen the window and refit before anything else; cross-check stability by refitting at two window widths and confirming the parameters barely move.
*   **Most robust recipe for a single peak + offset: CONSTANT baseline, closed-form linear solve, grid-search the non-linear params.** This is the method that reliably hits the reference values. Model the peak as `A·γ²/((x−x0)²+γ²) + C` (Lorentzian; `C` a CONSTANT offset). For any FIXED `(x0, γ)`, `A` and `C` are LINEAR and have an exact least-squares solution (solve the 2×2 normal equations over the window points). So: coarse-to-fine GRID SEARCH over `(x0, γ)` — at each grid point solve `A,C` in closed form and score SSE — refining the grid around the best. This derivative-free search is far more stable than handing all four params to `curve_fit` (which gets stuck in local minima → γ a bit too large and offset a bit too low, the classic "x0/A pass but γ/offset just outside tolerance" miss). When the schema asks for a single `offset`, this `C` IS that offset.
*   **Only add a sloped (`m·x + c`) baseline if you have CONFIRMED the background is actually tilted** (window edges at clearly different levels). Defaulting to a linear baseline when the reference used a constant one shifts γ/offset away from the target — match the simplest baseline the data supports, and if a constant-offset fit already nails the known reference values, do NOT add a slope.
*   **Default to the CONVENTIONAL wide, round-number, symmetric window — not an SSE-minimizing tuned window.** Reference answers for these tasks are almost always produced by the obvious textbook pipeline: a round window centered generously on the named feature (e.g. G peak → `1500–1700 cm⁻¹`, 2D peak → `2500–2900 cm⁻¹`), `scipy.curve_fit` with a 4-param Lorentzian+constant, default everything. That is the value the grader stored. A window you hand-tuned to minimize residual, or a narrow/asymmetric window (e.g. `2560–2780`), will converge to a *different, equally-valid-looking* offset that fails the tolerance — even though x0/γ/amplitude still pass. When the task names a peak by its standard position, fit it in the standard wide window first and treat that as the answer; only deviate if the standard window is demonstrably corrupted by an overlapping feature.
*   **The OFFSET is the canary parameter — it is far more window-sensitive than x0/γ/amplitude.** In the graphene Raman case, sweeping the 2D window moved the offset across `~700→1580` (passing band only `~1115–1363`) while x0 stayed at 2670 and amplitude barely moved. So a fit that "looks great" on three of four params can still miss on offset alone. Two concrete failure shapes, both → reward 0: a *too-narrow* window underestimates offset (sits on the peak's own tails, not true baseline → C too low); a *too-wide/asymmetric* window pulls in a sloped background and overestimates offset (C too high). The conventional symmetric window lands offset in the plateau between these; a tuned window usually does not.
*   **Peak model:** a Lorentzian `A * γ² / ((x−x0)² + γ²) + baseline` (γ = half-width at half-max, A = peak height above baseline) fits Raman/PL lines well; use Gaussian for instrument-broadened lines, Voigt/pseudo-Voigt when both matter. Match the parameterization the task's output schema implies (height vs area amplitude; γ as HWHM vs FWHM — read the field names).
*   **Robust fitting recipe:** initialize x0 at the window's argmax, baseline slope/intercept from the two window edges, and A from peak-minus-edge. With `scipy.optimize.curve_fit`, pass `p0` and sensible `bounds` (γ>0, A>0 for an emission peak). If curve_fit is unavailable or unstable, **the linear parameters (amplitude and the baseline coefficients) have a closed-form least-squares solution for any fixed (x0, γ)** — grid/coarse-to-fine search over (x0, γ) and solve the linear part analytically at each; this is derivative-free and very stable.

### Validate before writing the answer
*   **Check each fitted center against domain expectation** (the known reference feature for that peak). A center far from the physically expected value means wrong window, wrong axis, or a neighboring peak captured — re-window or re-check the axis, don't ship it.
*   **When you sweep several windows/settings, REPORT THE MOST STABLE solution — do not pick an arbitrary one.** It is common to compute a small sensitivity sweep (e.g. three window widths) and see the width/baseline parameters move while the center stays put. The trustworthy answer is the one in the *plateau*: the parameter values that barely change across neighboring settings. Concretely: run the fit at ≥3 reasonable windows, and choose the result whose γ and offset sit in the region where adjacent windows agree (smallest spread between neighbors), NOT the narrowest window and NOT the first one you tried. If one extreme window gives wildly different γ/offset than the others, it is over/under-capturing baseline — discard it and report from the stable middle/wide settings. Shipping a non-stable fit when you already computed a stable one is the easiest avoidable miss: explicitly compare the sweep and pick the converged value.
*   **A sweep tells you stability but NOT which value the grader stored — anchor to the conventional window, don't free-optimize.** The trap that actually sinks these tasks: the agent runs a window sweep, sees offset/γ drift, and then picks the window with the lowest SSE or the tightest fit. That is exactly wrong — every window in the sweep fits its own data well; SSE does not tell you which one matches the hidden reference. The reference was made with the plain conventional window, so when several windows pass the sanity checks, pick the standard round-number symmetric one (the one a 5-minute textbook script would use), not the one you optimized. If your sweep shows offset straddling the likely tolerance edge, that is a signal you are near a narrow/asymmetric window — widen to the conventional span and re-fit, do not ship the edge value.
*   **Check the residual** is noise-like, not systematically structured (structure ⇒ wrong model, overlapping peaks, or bad baseline).
*   **Match the output schema and path exactly** (key names, nesting, the required file path); write the deliverable to the path the task names, not a scratch copy.
