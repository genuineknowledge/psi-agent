### When the signature isn't pinned, design a TOLERANT interface
A grader for a sampler usually calls your function its OWN way, and the prompt
often does NOT specify the exact signature — so a rigidly-guessed signature is the
top cause of an otherwise-correct sampler failing.
*   **Accept the domain/bounds in multiple forms.** A grader may pass the support as a single vector `c(lo, hi)` (one positional arg), OR as two separate `lower`/`upper` args, OR via named args, OR omit it (defaulting to the density's natural support). Detect and normalize all of these — e.g. if the bounds argument has length 2, unpack it as the `(lo, hi)` pair; if two scalars are given, use them. Do NOT hard-require each bound to be a length-1 scalar — that rejects the common "pass the domain as one `c(lo,hi)` vector" call shape.
*   **Support named arguments and be order-tolerant.** Graders commonly call with only some args named (e.g. the density and `n` by name, domain omitted or positional). Use named params with sensible defaults and infer roles by TYPE (a function = the density, a length-2 numeric = the domain, a scalar = the count `n`) rather than betting on one fixed positional order.
*   **Validate to REJECT illegal but ADMIT all legal inputs.** Input checks should catch genuinely bad input (negative/zero/non-integer n, lower≥upper, non-function density) WITHOUT excluding standard legal usage — unbounded support (`-Inf`/`Inf`), a vector domain, or omitted bounds. An over-strict validator that throws on the grader's normal call is indistinguishable from a broken sampler. Mirror the conventions of the language's standard package for this algorithm (look up the canonical `ars`/sampler signature) rather than inventing one.
*   **Proven tolerant shape (R, generalizes):** make the domain ONE optional vector arg with an unbounded default, and resolve which argument is the density vs the count by TYPE at the top of the body, so any positional order the grader uses works:
    ```r
    ars <- function(a, b, domain = c(-Inf, Inf), n = NULL, ...) {
      # figure out density & count regardless of order
      args <- list(a, b)
      f   <- args[[ which(vapply(args, is.function, logical(1)))[1] ]]
      num <- args[[ which(vapply(args, function(x) is.numeric(x) && length(x)==1, logical(1)))[1] ]]
      if (is.null(n)) n <- num
      if (length(domain) == 2) { lower <- domain[1]; upper <- domain[2] }
      ...
    }
    ```
    This accepts `ars(f, c(lo,hi), n=N)`, `ars(n=N, f=density)`, `ars(N, density)`, etc. The exact mechanism matters less than the principle: detect roles by type, treat the domain as a length-2 vector with an `c(-Inf,Inf)` default, never hard-fail on a 2-element domain.

### Core sampler correctness (ARS / rejection family)
*   **Work in log space.** ARS builds piecewise-linear upper (and lower/squeezing) hulls of `log f`; operate on `logf` to avoid overflow and to make concavity checks linear.
*   **Check log-concavity during sampling, as required.** Verify the tangent/secant hull property holds at the abscissae; if a proposed point reveals `log f` is not concave (an upper-hull violation), raise the specified error rather than returning biased samples.
*   **Envelope + squeeze + rejection loop:** sample from the exponential-of-piecewise-linear upper hull, accept/reject against the squeezing lower bound first (cheap) then the exact density, and **add each evaluated point to refine the hull** so the envelope tightens adaptively.
*   **Handle (semi-)infinite support:** initialize abscissae using finiteness of the bounds; for an unbounded side, pick starting points where the log-density has positive/negative slope so the hull is proper.

### Formal stochastic testing (the spec usually demands it)
*   **Test against known truth with tolerances, since output is random:** draw a large sample from a standard distribution (normal, exponential) and assert the sample mean/sd fall within a tolerance band around the distribution's known mean/sd — not exact equality. Size the band to be loose enough for sampling noise at your sample size but tight enough to catch a wrong distribution.
*   **Emit results in the exact format the spec asks** (`"TEST_NAME: PASS"`/`"FAIL"` plus mean/sd), write any required sample files to the named paths, and keep the code modular (separate hull-construction, sampling, validation) as graders often check structure.
*   **Self-test the grader's likely call shapes before finishing:** invoke your sampler the way a grader plausibly would (vector domain, named args, a standard density) and confirm it returns samples rather than erroring on the interface.

### Get the file written — don't let environment setup eat the budget
*   **The deliverable (e.g. `/app/ars.R`) is most of the score; reaching it is the failure point.** A common 0-reward pattern: spend the whole budget installing the language runtime (R) and never write the file. Install in ONE foreground command (`apt-get update && DEBIAN_FRONTEND=noninteractive apt-get install -y r-base-core && echo __DONE__`), and do not be fooled if the terminal looks blank/frozen while a long foreground install runs — that is the install holding the foreground, not a hang; wait for the `__DONE__` marker, then immediately write the file.
*   **Write the whole implementation in ONE shot via base64** (`printf '%s' '<b64>' | base64 -d > /app/ars.R`) — a multi-hundred-line sampler written line-by-line will not finish in budget, and a heredoc can hang the terminal. Compose the complete file, base64 it, write it in a single command, then `source` it and run the tests.

### Porting a Bayesian model (rstan → PyStan/etc.) and running it
*   **The provided model script IS the spec — copy the Stan code and the EXACT sampling hyperparameters.** For "convert this rstan model to PyStan" tasks, read the R script and reproduce: the full Stan program (functions/data/parameters/model/generated quantities blocks verbatim), and every sampling control — `chains`, `iter`/`warmup` (PyStan splits these into `num_samples`+`num_warmup`; remember PyStan's `num_samples` = R's `iter - warmup`), `thin`→`num_thin`, `seed`/`random_seed`, and the `control=list(adapt_delta=..., max_treedepth=..., adapt_gamma/kappa/t0, init_buffer/term_buffer/window)` → PyStan kwargs (`delta`, `max_depth`, `gamma`, `kappa`, `t0`, `init_buffer`, `term_buffer`, `window`). The grader compares your posterior means to the R run, so a mismatched warmup/thin/delta changes the answer. Honor explicit constraints (e.g. "set `random_seed=1`", "must use PyStan 3.10.0", "do NOT use cmdstanpy / do NOT install R").
*   **Write the output files in the exact shape the grader parses** — e.g. numeric-only CSVs, one value per row for vectors, no header. Extract posterior means with `numpy.mean` over the drawn samples per parameter.
*   **CRITICAL perf fix: keep big intermediates OUT of `transformed parameters` — declare them as LOCALS in the `model` block.** Stan STORES and serializes every `transformed parameters` value for every retained draw. For a GP with an N×N covariance `K` and its `cholesky_decompose` `L_K` (e.g. N=200 → 40,000 doubles × 2 matrices), keeping them in `transformed parameters` writes BILLIONS of numbers through PyStan/httpstan over 4 chains × thousands of draws — turning a ~5-minute run into one that never finishes in budget (real failure: sampling crawled 0%→55% over 100+ turns, timed out). A faithful rstan→PyStan port that copies the R model's `transformed parameters { K; L_K; }` verbatim is CORRECT but pathologically slow under httpstan. Fix: move `mu`, `K`, `L_K` (anything you don't need reported, used only to build the likelihood) into the `model` block as local variables — the posterior is identical, but nothing huge is stored. Only keep in `transformed parameters` what the task actually needs in the output. The instruction's "optional: optimize the Stan model for memory efficiency" is pointing exactly here; treat it as mandatory for any large-matrix GP/hierarchical model.
*   **If you need the parameter draws but the model is still heavy, run chains concurrently** (PyStan samples chains in parallel by default; ensure enough effective parallelism) and time-box: estimate (warmup+samples) × per-step Cholesky cost vs the budget BEFORE launching, and apply the model-block-locals fix first — it is usually the single biggest win.

### Running and MONITORING long sampling jobs (the silent budget-killer)
A GP / hierarchical MCMC with `multi_normal_cholesky` and an N×N `cholesky_decompose` recomputed every leapfrog step, at `max_treedepth=14` and 4 chains on 1 CPU, can run for many minutes to hours. The fatal pattern (real failure: 150+ min of blind polling, trial timed out at 0): launch the run piped through `| tail -30`, see a BLANK screen (tail buffers until the process exits), and then poll every turn assuming "it's just slow" — never knowing if it's compiling, sampling, hung, or already dead.
*   **Never pipe a long job through `tail`/`head`** — that hides all progress until exit. Redirect to a logfile and poll THAT: `nohup python3 /app/run.py > /app/run.log 2>&1 &` then `tail -n 30 /app/run.log` each check. PyStan/Stan print `Building...`, then per-chain `Sampling: NN%` progress to the log — you need to SEE it.
*   **Read the FIRST error instead of polling through it.** A blank screen after launch often means the process already CRASHED (e.g. `stan.build` → `Building...` then a traceback: `gcc`/`build-essential` missing). Check the log / exit status within a turn or two of launching; don't interpret silence as "still working." Install `build-essential` BEFORE the first build if a C++ compiler may be absent.
*   **Estimate runtime against the budget and set a hard deadline.** Before launching, sanity-check that 4 chains × (warmup+samples) × an N×N Cholesky per step can finish inside the task timeout; run with a `timeout`, and if it won't fit, run chains in parallel (`STAN_NUM_THREADS`, or background the 4 chains) and/or cut wall-clock by sampling chains concurrently — but do NOT silently change `iter`/`warmup`/`thin`/`seed`, which would change the graded posterior means. If a single foreground run is genuinely too slow, background it early and spend intervening turns on nothing else that matters, checking the log on a schedule with a decision point (kill + diagnose) rather than open-ended polling.
*   **A poll that shows no new output is information: after 2-3 empty polls, actively probe** — `ps aux | grep python`, check the logfile size is growing, check exit code — rather than emitting another "continue polling" turn. Open-ended polling produces nothing and burns the whole budget.
